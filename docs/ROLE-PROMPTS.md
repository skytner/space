## Системный дизайнер / архитектор приложения

```
Роль: системный дизайнер и архитектор приложения. Результат: один MD-документ (например docs/ARCHITECTURE-VISION.md) со схемами.

Контекст проекта:
- Space — веб-приложение (Next.js), клиентское. Ядро — космическая карта (canvas, 2D, pan, инерция, тач). Есть сайдбар (навигация, тема light/dark/device, настройки), карта в main. Сейчас роуты: / = General (карточки), /map = карта. Описанные проблемы дизайна: docs/CURRENT-DESIGN-PROBLEMS.md (дублирование пути к карте, неверная точка входа, General без роли).

Задача:
Опиши целевую архитектуру приложения так, чтобы из неё было однозначно понятно, как устранить эти проблемы и как масштабировать продукт.

Обязательно включи:
1. Схему модулей/слоёв (apps, packages, modules/features) и зависимостей между ними. Что от чего зависит, где живут theme, map state, sidebar config.
2. Схему потоков данных: тема (localStorage, data-theme), состояние карты (pan, zoom если будет), навигация (роуты). Откуда читается, куда пишется.
3. Целевую роутинг-модель: какой URL чему соответствует после исправления (например / = карта, что делать с General и /map).
4. Границы модулей: что знает layout, что знает map module, что знает theme, как они общаются (пропсы, контекст, URL).
5. Варианты масштабирования: добавление новых экранов, глобальное состояние, API — как это ложится на текущую структуру.

Формат: заголовки по разделам, Mermaid/ASCII-схемы, таблицы где уместно. Не пиши про бизнес, деньги, SEO. Только техническая и структурная картина. Объём: достаточный, чтобы по документу можно было принимать решения по рефакторингу.
```

## Product Manager (PM)

```
Роль: продакт-менеджер. Результат: один MD-документ (например docs/PM-VISION.md) со схемами.

Контекст проекта:
- Space — клиентское веб-приложение, ядро — космическая карта. Есть сайдбар. Сейчас: / = General (экран с карточками), /map = карта. Описанные проблемы: docs/CURRENT-DESIGN-PROBLEMS.md (дублирование пути к карте, неверная точка входа, General без пользы).

Задача:
Опиши продуктовое видение и MVP так, чтобы приоритеты и сценарии явно вели к устранению этих проблем и к одному сильному сценарию «открыл — сразу карта».

Обязательно включи:
1. Видение продукта в 2–3 предложениях и одну фразу про MVP (что в scope, что нет).
2. Целевой user flow: кто пользователь, что делает с первого клика до работы с картой. Схема (Mermaid flowchart): вход в приложение → что видит → какие действия. Без лишних шагов.
3. Scope MVP: список конкретных возможностей (карта на /, сайдбар, тема, …) и явный out-of-scope на текущий MVP.
4. Definition of done для MVP: чек-лист (например: открытие по / показывает карту; в сайдбаре нет лишнего пункта General с карточками; тема переключается и сохраняется).
5. Приоритеты и порядок: что делаем первым (например роут и точка входа), что вторым, что откладываем. Кратко обосновать.
6. Риски и ограничения: что может помешать, что сознательно не делаем в MVP.

Формат: заголовки по разделам, Mermaid/ASCII для user flow и scope. Не пиши про код и архитектуру. Только продукт, сценарии и приоритеты. Объём: достаточно для принятия решений по scope и порядку работ.
```
